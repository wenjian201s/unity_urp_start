基于Unity URP与Compute Shader的高性能自适应曝光与局部色调映射系统实现研究
摘要
在现代基于物理的渲染（PBR）管线中，高动态范围（HDR）技术能够真实还原现实世界的光照强度，但直接将HDR图像输出至低动态范围（LDR）显示设备会导致画面过曝或欠曝。传统的全局自动曝光算法在面对极端高对比度场景（如逆光、窗口入口）时，常导致局部细节丢失。本文提出并详细解析了一套基于Unity Universal Render Pipeline (URP) 与Compute Shader构建的自适应曝光系统。该系统通过屏幕空间像素降维聚合、对数空间域转换、时间域滤波平滑，以及引入基于双边滤波的局部曝光算法，实现了既符合人眼视觉适应规律，又能保留高对比度边缘细节的曝光控制。本文同时针对Unity 2022.3版本底层的CommandBuffer架构与内存管理机制进行了深度适配与优化分析。

关键词：自动曝光；局部色调映射；Compute Shader；URP；双边滤波；时间滤波；对数空间

第一章 引言
1.1 研究背景与问题提出
在传统的图形渲染管线中，色彩通常被限制在0到1的线性空间或sRGB空间内。随着基于物理的渲染（PBR）成为行业标准，光源的强度（单位坎德拉/流明）被赋予了真实的物理量级。太阳的照度可能高达100,000 lux，而室内的照度可能仅有几百lux。这种巨大的动态范围远远超出了传统显示器的显示能力。

为了将HDR映射到LDR，必须引入色调映射与曝光控制。全局自动曝光通过计算整个屏幕的平均亮度来决定一个统一的曝光乘数。然而，全局算法存在一个致命缺陷：“剪影效应”或“局部信息丢失”。当摄像机位于昏暗的室内看向明亮的室外时，全局平均亮度会被室外高光拉高，导致系统强制压低整个画面的亮度，最终室内原本可见的细节沦为死黑。人眼的视觉系统并非如此工作，人眼具有“局部适应”能力，能够同时在视网膜的不同区域维持不同的感光灵敏度。

第三章 系统架构设计与C#宿主程序实现 (Unity 2022.3)
在Unity URP中，自定义后处理无法像旧版MonoBehaviour那样直接挂在摄像机上，必须遵循严格的 VolumeComponent -> ScriptableRendererFeature -> ScriptableRenderPass 三级架构。

3.1 参数抽象层：VolumeComponent
AutoExposure 类继承自 VolumeComponent。URP的Volume系统允许参数在场景中叠加、插值和优先级覆盖。

csharp

[VolumeComponentMenu("Custom/AutoExposure")]
public class AutoExposure : VolumeComponent
{
    public BoolParameter localExposure = new BoolParameter(false);
    public ClampedFloatParameter blurredLumBlend = new ClampedFloatParameter(0.5f, 0f, 1f);
}
技术解析：不能使用原生的 bool 或 float，必须使用 BoolParameter 等包装类。因为URP框架需要在不同的Volume之间对参数进行插值混合（例如从室内Volume走到室外Volume，参数需平滑过渡），原生类型无法实现这种多态状态管理。

3.2 管线注入层：ScriptableRendererFeature
LocalExposureRendererFeature 是管线与外界的接口。

csharp

public override void AddRenderPasses(ScriptableRenderer renderer, ref RenderingData renderingData)
{
    renderer.EnqueuePass(localExposureRenderPass);
}
protected override void Dispose(bool disposing)
{
    localExposureRenderPass?.Dispose();
}
技术解析：AddRenderPasses 每帧被URP调用，将Pass压入执行队列。极其重要的一点是 Dispose 的重写。Unity 2022.3 没有 Render Graph 的自动内存追踪，如果在 Feature 被移除或管线重建时不调用 Pass 的 Dispose，会导致 GPU 显存泄漏。

3.3 执行与资源管理层：ScriptableRenderPass (核心攻坚)
这是最复杂的部分，涉及跨语言内存通信与底层API调度。

3.3.1 跨平台内存对齐：[StructLayout]
C# 运行在CLR上，默认会为了CPU缓存行对齐而对结构体字段进行重排（如将一个float塞到两个short中间的空隙）。GPU显存（HLSL的 StructuredBuffer）则是严格按照声明顺序排列的。

csharp

[StructLayout(LayoutKind.Sequential)]
public struct RWParameters
{
    public uint importance; public uint luminance;
    public float historyEV; public float exposure;
}
如果不加 LayoutKind.Sequential，C#传给GPU的数组会发生“字节错位”，GPU读出的 historyEV 可能其实是C#里 luminance 的一部分二进制位，导致计算出漫天乱飞的NaN颜色。

3.3.2 ComputeBuffer 的精确映射
csharp

rwParameterBuffer = new ComputeBuffer(1, Marshal.SizeOf(typeof(RWParameters)), ComputeBufferType.Structured);
技术解析：Marshal.SizeOf 在运行时反射计算结构体的精确字节数（考虑了平台指针大小和对齐规则）。必须保证这里的字节数与HLSL中 struct RWParameters { ... } 占用的字节数绝对一致（本文中为 16 字节：4+4+4+4）。

3.3.3 RTHandle 与 UAV 权限 (Unity 2022.3 特有机制)
Compute Shader 要对纹理进行随机读写（_ScreenTexture[id.xy] = xxx），要求纹理必须具有 UAV（Unordered Access View）权限。URP默认的相机渲染目标是不具备该权限的。

csharp

public override void OnCameraSetup(CommandBuffer cmd, ref RenderingData renderingData)
{
    var descriptor = renderingData.cameraData.cameraTargetDescriptor;
    descriptor.msaaSamples = 1; // UAV 不支持 MSAA
    descriptor.enableRandomWrite = true; // 开启随机写入
    descriptor.depthStencilFormat = GraphicsFormat.None; // 纯计算不需要深度

    descriptor.colorFormat = RenderTextureFormat.ARGBHalf;
    RenderingUtils.ReAllocateIfNeeded(ref screenTexture, descriptor, FilterMode.Point, TextureWrapMode.Clamp, name: "_ScreenTex");
    
    descriptor.colorFormat = RenderTextureFormat.RHalf; // 降维至单通道
    RenderingUtils.ReAllocateIfNeeded(ref logLuminance, descriptor, ...);
    // ...
}
技术深度：

ReAllocateIfNeeded 是2022.3极其关键的API。它不仅会分配内存，还会检测如果分辨率没变，就复用上一帧的RTHandle，避免每帧 new RenderTexture 造成的巨大GC压力和性能峰值。
带宽极限优化：对于亮度图，抛弃了 ARGBHalf（64位/像素），改用 RHalf（16位/像素）。对于一张 1080p 的纹理，这直接节省了 
1920×1080×(8−2)=12.4MB
 的显存带宽占用。在后续的双边滤波中，读取单通道数据的速度提升了数倍。

csharp

public override void Execute(ScriptableRenderContext context, ref RenderingData renderingData)
{
    CommandBuffer cmd = CommandBufferPool.Get("Local Exposure Compute");
    // 1. 拷贝原生画面到带UAV的中转图
    cmd.Blit(renderingData.cameraData.renderer.cameraColorTargetHandle, screenTexture);
    
    // 2. 调度Compute Shader...
    
    // 3. 接回主管线 (极易错漏的一步)

避坑指南：最后一步 cmd.Blit 是绝对不可省略的。我们在自定义Pass中修改的只是 screenTexture，URP后续的UI渲染、后处理栈依然在读取 cameraColorTargetHandle。如果不把图“还”回去，屏幕将永远显示未曝光的旧图。

第四章 GPU并行算法实现与数学模型
本章深入HLSL层面，解析五个核心Kernel的底层逻辑。

4.1 Kernel 1: 屏幕空间降维聚合
将数百万像素的颜色信息，压缩为两个整数。

hlsl

[numthreads(16, 16, 1)]
void AccumulateLuminance(uint3 id : SV_DispatchThreadID, uint groupIndex : SV_GroupIndex)
{
    if (!any(id.xy)) { /* 清零操作 */ }
    
    // 中心权重模型
    half2 uv = id.xy / _ScreenSize.xy;
    float importance = pow(saturate(1.0 - (distance(uv, 0.5) * 2.0)), 2);

    float luminance = dot(_ScreenTexture[id.xy].rgb, float3(0.2126, 0.7152, 0.0722));
    
    // 量子化与原子累加
    uint imp = (uint) (100.0 * importance);
    uint lum = (uint) (100.0 * luminance * importance);
    InterlockedAdd(_RWParameters[0].importance, imp);
    InterlockedAdd(_RWParameters[0].luminance, lum);
    
    _LogLuminance[id.xy] = log2(max(luminance, 1e-5));
}
算法解析：

单线程初始化：!any(id.xy) 是一种极其优雅的并行编程技巧，确保数以万计的线程中，只有坐标为 (0,0) 的那一个线程执行清零，避免了多线程同时写0的竞争开销。
空间衰减权重：pow(..., 2) 是二次衰减曲线。它比线性衰减更极端地剥夺屏幕边缘的投票权。这符合摄影中的“中心构图”法则，避免画面上方的一小块天空主导了整张图的曝光。
整数原子操作的秘密：为什么不直接用 InterlockedAdd 操作浮点数？因为在早期的GPU硬件（如GTX 10系列及部分移动端GPU）上，对浮点数的原子加法要么不支持，要么性能极差（需要在ALU外模拟锁机制）。通过乘以100转为 uint，利用硬件原生的整数原子加法指令，实现了跨所有GPU架构的绝对安全与高性能。100的精度对于亮度统计完全足够。
4.2 Kernel 2: 单线程串行决策与平滑
hlsl

[numthreads(1, 1, 1)]
void ComputeTargetEV(uint3 id : SV_DispatchThreadID)
{
    float avgLuminance = (0.01 * lum) / max(0.01 * imp, 1e-5);
    float avgEV = log2(max(avgLuminance, 1e-5) * 8.0);
    
    float t = min(_RParameters[0].deltaTime, 0.033);
    float targetEV = _RWParameters[0].historyEV + (avgEV - _RWParameters[0].historyEV) * t;
    
    _RWParameters[0].historyEV = targetEV;
    _RWParameters[0].exposure = 1.0 / pow(2.0, targetEV);
}
架构解析：这里刻意使用了 [numthreads(1, 1, 1)]。在高度并行的GPU上开启1个线程看似浪费，实则是同步屏障的替代方案。前一个Kernel有成千上万个线程在写 _RWParameters，如果不加屏障直接执行后续逻辑，读取到的数据可能只被部分线程更新。将其设为1个线程的独立Dispatch，利用了GPU硬件保证的前后依赖关系，确保在此Kernel开始时，前一个Kernel的所有原子写入绝对已经落盘。

### 4.5 Kernel 5: 动态变体与最终合成
```hlsl
#pragma multi_compile_local __ LOCAL_EXPOSURE

float localExposure = 1.0;
#ifdef LOCAL_EXPOSURE
    float bilateralLum = exp2(_BilateralLogLuminance[id.xy]);
    float gaussianLum = exp2(_GaussianLogLuminance[id.xy]);

技术深度：

Shader变体：multi_compile_local 让编译器生成两个版本的Shader。当C#关闭局部曝光时，GPU完全不执行那些耗时的数学运算，实现性能的“零开销”。
空间转换：exp2 将对数空间的模糊结果还原回线性空间的亮度。
艺术上的妥协 (0.5系数)：理论上，将局部EV完全拉平到全局EV（系数为1.0），能看清所有暗部细节。但这会丧失画面的光影对比，画面显得“灰、平”。乘以0.5系数意味着“只补偿一半的亮度差”，在细节与电影感之间找到了完美的平衡点。
混合度控制：lerp 混合了双边与高斯结果。纯双边可能保留过多噪点，纯高斯可能产生光晕。美术可以通过Volume滑块自由调节。
第五章 性能优化与工程陷阱排查
在实际工程落地中，这套系统遇到了诸多引擎底层机制的阻碍，以下是排查与优化的记录。

5.1 “黑屏”与UAV权限限制
现象：Compute Shader 执行无报错，但画面全黑。
排查：通过 RenderDoc 抓帧，发现 _ScreenTexture 未被成功写入。
根因：URP默认分配的Builtin RenderTexture没有 DX11_UNORDERED_ACCESS 标志。Compute Shader 的 RWTexture2D 写入操作被DirectX/Vulkan底层API静默拒绝。
解决：必须在C#端重新实例化一张 enableRandomWrite = true 的RT，并增加一次 Blit 拷贝。这是在URP中做Compute后处理的固定开销。
5.2 “数据错乱”与内存对齐
现象：Volume面板调参时，画面亮度发生毫无规律的剧烈跳变。
排查：在HLSL中硬编码返回 _RWParameters[0].historyEV，发现其值为随机的极大浮点数。
根因：C# 的 struct RParameters 包含两个 float（8字节），由于未加 [StructLayout]，CLR可能将其对齐为16字节。而HLSL严格按照8字节读取，导致下一次读取时错位到了下一个缓冲区的数据。
解决：强制加上 [StructLayout(LayoutKind.Sequential)]，并使用 Marshal.SizeOf 动态获取字节，杜绝人为数错位数的风险。
5.3 上下文绑定错误
现象：局部曝光开启后，画面边缘出现强烈的“反向光晕”。
排查：对比C#端的 SetComputeTextureParam 与HLSL的变量声明。
根因：在传入 ApplyExposure Kernel 时，由于变量名相似，C#端将 _GaussianLogLuminance 错误地绑定到了HLSL中声明的 _BilateralLogLuminance 槽位上。
解决：严格规范字符串硬编码，或在C#端使用 Shader.PropertyToID 获取ID。在重构时，必须保证C#的赋值顺序与HLSL的声明顺序绝对一一对应。

第六章 架构演进：从 Unity 6000 到 2022.3 的降级哲学
本文原始代码基于 Unity 6000 的 Render Graph API，在实际项目需求中，被降级适配至 Unity 2022.3。这反映了两种截然不同的管线设计哲学。

6.1 Render Graph (Unity 6000) 的“推演式”哲学
Render Graph 将渲染过程抽象为一张“图”。开发者只负责声明节点（Pass）和依赖（UseTexture），不负责分配资源。

优点：中间纹理（如本文的 _GaussianTempBuffer）的生命周期被完全自动化管理。如果两个Pass没有依赖关系，Render Graph会使用内存别名技术，让两张纹理共享同一块显存（Transient Resource Allocation）。
缺点：学习曲线陡峭；Lambda闭包捕获容易导致难以排查的引用错误；对于动态分支（如本文的 if (volume.localExposure)），在Graph中处理较为繁琐。
6.2 CommandBuffer (Unity 2022.3) 的“命令式”哲学
开发者必须像一个微观管理者，明确指出每一张图的创建、格式、复用与释放。

痛点：本文中的 ReAllocateIfNeeded 是对这种痛点的最佳缓解。如果不使用它，开发者必须写大量的 if (rt != null && (rt.width != w)) rt.Release(); 样板代码。
陷阱：如果在 Execute 中直接 new RenderTexture，会导致每帧产生数兆的GC Alloc，引发游戏卡顿。如果不写 Dispose，在编辑器切换场景时会爆显存。
通过本次降级实践，深刻认识到：底层API虽然繁琐，但给予了开发者对显存绝对的控制权。在处理复杂的、带有大量中间缓冲的 Compute Pipeline 时，2022.3 的显式管理反而更容易进行针对性的池化优化。

第七章 结论与未来展望
7.1 结论
本文详细解析并实现了一套工业级的自动曝光系统。该系统在理论上融合了摄影学EV模型、韦伯-费希纳对数感知定律；在算法上采用了整数原子聚合、时间域低通滤波、对数域双边保边滤波以及分离式高斯降维；在工程上，克服了跨语言内存对齐、UAV权限限制、显存带宽瓶颈以及复杂的URP管线注入难题。

特别是局部曝光模块的引入，以相对可控的性能开销，彻底解决了传统全局色调映射无法兼顾高光与暗部细节的顽疾，极大地提升了画面的电影感与真实感。

7.2 未来展望
尽管当前系统已具备较高的完备性，但仍存在进一步优化的空间：

Bilateral Grid 算法升级：当前双边滤波复杂度为

O(N)
，从而支持更大半径的模糊。
历史帧缓存复用：当前的时间滤波仅复用了EV标量。可以将上一帧的模糊亮度图进行运动向量补偿后复用，与当前帧的模糊结果混合，进一步消除由于降噪/模糊带来的画面抖动。
自适应动态分辨率结合：在性能极度受限时，可以将亮度统计和模糊计算降至 Quarter 分辨率（1/4尺寸）进行，然后上采样，利用人眼对低频亮度不敏感的特性，实现极致的性能榨取。
本技术方案不仅在 Unity URP 中具有极高的实用价值，其底层的数学模型与并行计算思想，亦可无缝迁移至 Unreal Engine 的 Compute Shader 或纯底层的 Vulkan/DirectX 12 现代图形API中。
